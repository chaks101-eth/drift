# Rating Enricher Agent

You ensure generated itinerary items are enriched with real Google Places data — ratings, GPS coordinates, photos, and addresses.

## The Problem

Eval data shows `ratingQuality` at 56/100 — items generated by the LLM often lack:
- `metadata.rating` (Google Places star rating)
- `metadata.reviewCount`
- `metadata.lat` / `metadata.lng` (GPS)
- `image_url` (real venue photo)

## Root Cause

The generation pipeline (`src/app/api/ai/generate/route.ts`) calls `applyPlaceData()` post-generation, but:
1. Some items may not match on Google Places (name mismatch)
2. The enrichment may be skipped on error
3. Items from LLM fallback (non-catalog) may bypass enrichment entirely

## Investigation Steps

1. Read `src/app/api/ai/generate/route.ts` — find where `applyPlaceData` or `batchGetPlaceData` is called
2. Read `src/lib/google-places-photos.ts` — understand `applyPlaceData()` behavior
3. Check: what % of generated items actually get enriched?
4. Check: are there error paths that skip enrichment?

## Audit Command

```bash
# Check how many items have ratings across recent trips
ADMIN_SECRET=$(grep ADMIN_SECRET .env.local | cut -d= -f2)
SERVICE_KEY=$(grep SUPABASE_SERVICE_ROLE_KEY .env.local | cut -d= -f2)
SUPABASE_URL=$(grep NEXT_PUBLIC_SUPABASE_URL .env.local | cut -d= -f2)

# Count items with/without ratings from recent trips
curl -s "$SUPABASE_URL/rest/v1/itinerary_items?select=name,category,metadata&order=created_at.desc&limit=50" \
  -H "apikey: $SERVICE_KEY" \
  -H "Authorization: Bearer $SERVICE_KEY"
```

## Fix Options

1. **Ensure applyPlaceData runs on ALL items** — no early returns on partial failure
2. **Retry failed enrichments** — if Google Places can't find "Fancy Restaurant Phuket", try "Fancy Restaurant, Phuket, Thailand"
3. **Fallback ratings** — if no Google Places match, use the catalog's stored rating
